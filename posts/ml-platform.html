推荐系统 算法工程 2025

大厂推荐系统实践

从 NYC Marketing 到推荐算法工程师的转型之路

引言

从纽约的市场营销工作转行到国内互联网大厂的推荐算法工程师，这段经历让我对技术和业务有了更全面的理解。 营销背景让我更关注用户体验和商业价值，而算法工程能力让我能够将这些洞察落地为实际的产品改进。

本文将分享我在大厂推荐系统工作中积累的实战经验， 包括召回、排序、特征工程、模型迭代以及 AB Test 等核心环节的实践心得。

推荐系统架构概览

一个典型的大厂推荐系统通常采用多阶段漏斗架构，从海量候选中逐步筛选出最终推荐结果： 

┌─────────────────────────────────────────────────────────┐
│                     用户请求层 (Request)                    │
│                    用户ID + 上下文信息                       │
├─────────────────────────────────────────────────────────┤
│                     召回层 (Recall)                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ 协同过滤  │ │ 向量检索  │ │ 图模型   │ │ 规则策略  │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │
│                   ↓ 千级候选集                              │
├─────────────────────────────────────────────────────────┤
│                     粗排层 (Coarse Ranking)                │
│              轻量级模型快速打分筛选                          │
│                   ↓ 百级候选集                              │
├─────────────────────────────────────────────────────────┤
│                     精排层 (Fine Ranking)                  │

核心模块详解

1. 召回层 (Recall)

召回层的目标是从海量物料中快速筛选出千级别的候选集。我们采用了多路召回策略：

  • i2i (Item-to-Item)：基于物品的协同过滤，计算物品相似度
  • u2i (User-to-Item)：基于用户的协同过滤，找到相似用户喜欢的物品
  • 向量召回：使用双塔模型将用户和物品编码为向量，通过 ANN 快速检索
  • 图模型：利用 Graph Embedding 捕捉高阶连接关系
实践经验：召回的覆盖率比准确率更重要。宁可错召，不可漏召。 精排层有足够的能力将不相关的物品过滤掉。

2. 特征工程

特征工程是推荐系统的核心竞争力。我们将特征分为几个维度：

特征类别 示例 处理方式
用户画像 年龄、性别、地域、兴趣标签 One-hot / Embedding
物品特征 类别、标签、创作者、发布时间 Embedding / 统计特征
上下文 时间、设备、位置、网络 离散化 / 交叉特征
行为序列 点击、收藏、分享历史 Sequence Modeling
交叉特征 用户-物品匹配度 FM / DeepFM

实时特征 vs 离线特征

特征的新鲜度直接影响推荐效果。我们采用 Lambda 架构处理特征：

  • 离线特征：T+1 更新，用于用户画像、物品统计等变化较慢的特征
  • 实时特征：秒级延迟，用于用户最近行为、当前会话上下文等

3. 模型迭代

从传统的 LR 到深度模型，推荐系统的模型经历了多代演进：

模型演进路径

LR (2015) 
  ↓
FM / FFM (2016) - 自动二阶交叉
  ↓
Wide & Deep (2017) - 记忆 + 泛化
  ↓
DeepFM (2018) - 端到端学习交叉
  ↓
DIN (2019) - 注意力机制建模用户兴趣
  ↓
DIEN (2020) - 序列建模 + 兴趣演化
  ↓
MMoE / PLE (2021+) - 多任务学习

4. AB Test 实践

AB Test 是推荐系统迭代的核心驱动力。我们的 AB Test 体系包括：

  • 分流：基于用户 ID 哈希，确保用户每次进入同一实验组
  • 互斥域：不同层级的实验相互独立，避免相互干扰
  • 指标：
    • 北极星指标：人均使用时长、DAU 等
    • 过程指标：点击率、完播率、收藏率等
    • 健康指标：多样性、新鲜度等
常见坑点：实验组和对照组的样本量要足够大，运行时间要足够长（至少一周覆盖完整用户周期）。 小流量实验可能会因为样本偏差得出错误结论。

从营销到算法的跨界思考

1. 用户视角

Marketing 背景让我在做推荐算法时始终关注用户真实需求， 而不是单纯追求技术指标。一个 CTR 很高的推荐，如果让用户感到厌烦或疲劳， 长期来看是损害用户体验的。

2. 数据驱动

Marketing 教会我如何设计实验、解读数据。这种思维方式在 AB Test 和模型评估中非常有价值。 不只是看指标涨跌，更要理解为什么会涨跌，背后的用户行为逻辑是什么。

3. 业务理解

推荐系统不只是技术问题，更是业务问题。理解商业目标、用户生命周期价值、 内容生态健康度等，才能做出真正有价值的推荐策略。

总结

推荐系统是一个复杂的工程问题，需要在技术深度和业务理解之间找到平衡。 从营销转型的经历让我更能理解"为什么做"，而不仅仅是"怎么做"。

未来，随着大模型和 Agent 技术的发展，推荐系统可能会有新的范式。 但核心始终不变：理解用户，创造价值。
